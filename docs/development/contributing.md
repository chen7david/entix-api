# Contributing to Entix API

Thank you for your interest in contributing to Entix API! This document provides guidelines and instructions for contributing to the project.

## Code of Conduct

Please read and follow our [Code of Conduct](https://github.com/chen7david/entix-api/blob/main/CODE_OF_CONDUCT.md) to maintain a respectful and inclusive environment for everyone.

## Getting Started

1. **Fork the Repository**

   Fork the repository to your GitHub account.

2. **Clone the Repository**

   ```bash
   git clone https://github.com/your-username/entix-api.git
   cd entix-api
   ```

3. **Set Up Development Environment**

   Set up your development environment as described in the [Installation](/getting-started/installation.md) and [Development Environment](/getting-started/development.md) guides.

4. **Create a Branch**

   Create a new branch for your contribution:

   ```bash
   git checkout -b feature/your-feature-name
   ```

   Use a descriptive branch name that reflects your contribution.

## Development Workflow

1. **Make Changes**

   Make your changes to the codebase, following our [Coding Standards](/development/coding-standards.md).

2. **Write Tests**

   Add tests for your changes following our [Testing Guidelines](/development/testing.md).

3. **Run Tests**

   Ensure all tests pass:

   ```bash
   npm test
   ```

4. **Format Code**

   Ensure your code is properly formatted:

   ```bash
   npm run format
   ```

5. **Commit Changes**

   Commit your changes with a descriptive commit message:

   ```bash
   git commit -m "feat: add new feature"
   ```

   Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification for commit messages.

6. **Push Changes**

   Push your changes to your fork:

   ```bash
   git push origin feature/your-feature-name
   ```

7. **Create a Pull Request**

   Create a pull request from your fork to the main repository.

## Pull Request Guidelines

When creating a pull request:

1. **Fill out the Pull Request Template**

   Provide all the requested information in the pull request template.

2. **Link Related Issues**

   Link to any related issues using keywords like "Closes #123" or "Relates to #456".

3. **Describe Your Changes**

   Clearly describe what your changes do and why they're needed.

4. **Include Screenshots**

   If applicable, include screenshots or GIFs demonstrating the changes.

5. **Keep Pull Requests Focused**

   Each pull request should address a single concern or feature.

## Code Review Process

1. **Maintainers Review**

   A maintainer will review your pull request and may suggest changes.

2. **Address Feedback**

   If changes are requested, update your pull request by adding new commits.

3. **Merge**

   Once approved, a maintainer will merge your pull request.

## Types of Contributions

We welcome various types of contributions:

### Code Contributions

- Bug fixes
- New features
- Performance improvements
- Code refactoring

### Documentation Contributions

- Improving existing documentation
- Adding new documentation
- Fixing typos and clarifying language

### Other Contributions

- Reporting bugs
- Suggesting enhancements
- Improving tests
- Reviewing pull requests

## Reporting Bugs

When reporting bugs:

1. Use the bug report template
2. Include steps to reproduce
3. Describe expected vs. actual behavior
4. Include relevant information about your environment
5. Add screenshots if applicable

## Suggesting Enhancements

When suggesting enhancements:

1. Use the feature request template
2. Describe the enhancement clearly
3. Explain why it would be valuable
4. Suggest an implementation if possible

## Documentation Contributions

Documentation improvements are always welcome. Follow these guidelines:

1. Follow the [Documentation Guidelines](/development/documentation.md)
2. Ensure technical accuracy
3. Use clear and concise language
4. Include examples when helpful

## Community

Join our community channels:

- GitHub Discussions
- [GitHub Issues](https://github.com/chen7david/entix-api/issues)

## Attribution

Contributors will be acknowledged in our documentation and repository.
